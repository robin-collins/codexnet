protected script changed from SHA-256
`09bfdfd6d034c38882dfddf7cb648d64fc326fcf164f4c68ae49cb103eb2e526`
to the approved baseline
`1e502d15a5fe6663c97045660a31b5e4db1534e972d882839081edcea6f4659d`.
Only completed XML and containing directories gained `field-discovery` group
read/traverse access (`0640`/`0750`). Content hashes were unchanged by the
one-time migration; logs and all non-XML scan outputs remain `root:root 0600`.
Scan options, targets, interface derivation, retention, and scheduling were not

The live importer initially exposed a standards-compatibility defect because
nmap writes the identifier-free `<!DOCTYPE nmaprun>` declaration. The parser
now accepts only that exact declaration while continuing to reject external
identifiers, internal subsets, and entity declarations. Eight completed XML
artifacts then imported with zero errors; an immediate repeat and every
post-reboot timer pass skipped all eight with zero errors, proving idempotence.
The framework still supplies no active-scan timer. User/root cron and matching
active nmap systemd timers remain absent.
