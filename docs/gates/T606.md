# T606 security and privacy release review

Reviewed on 2026-07-15 on Debian 13 ARM64. No customer network, credential, or
customer data was used. The review covered source, Git history, dependencies,
configuration, process arguments, systemd units, staged installation paths,
SQLite, artifacts, backups, logs, JSON, diagrams, and unzipped DOCX packages.

## Dependency and source audit

- `pip check` passed in the exact locked environment.
- `pip-audit -r requirements-dev.lock --no-deps --disable-pip --format json`
  found one remaining advisory: Paramiko 4.0.0
  `PYSEC-2026-2858` / `CVE-2026-44405`. The audit reported no fix version.
  Netmiko 4.7.0 requires Paramiko `<5`, so the pin is retained temporarily.
  `NetmikoSessionFactory` explicitly disables `ssh-rsa` for host and public-key
  algorithms; tests prove the setting reaches every real session. This is an
  accepted lower risk owned by the project maintainer and must be re-audited
  on every dependency update.
- Pytest was upgraded from 8.4.1 to 9.0.3, resolving the development-only
  `PYSEC-2026-1845` finding. The second audit reported no pytest finding.
- The exact lock contains no Impacket, BloodHound, secretsdump, CrackMapExec,
  NetExec, or equivalent offensive package. Runtime/source scans found no
  password cracking, credential dumping, Kerberoasting, AS-REP roasting,
  attack-path collection, or brute-force implementation.
- Strong token/private-key patterns across the current tree and every Git
  revision returned zero findings. Generic secret-assignment candidates were
  confined to synthetic fixtures, redaction/parser tests, and denial rules.
  Git tracks no database, report, packet capture, backup, environment, key, or
  log artifact. Global ignore rules cover those formats.

No unresolved critical or high dependency/source issue remains.

## Privilege and filesystem audit

- Every service uses `field-discovery:field-discovery`, `UMask=0077`,
  `NoNewPrivileges=true`, `ProtectSystem=strict`, an explicit read-only config
  path, and only the owned `/var/lib/field-discovery` writable path.
- The passive observer alone receives `CAP_NET_RAW`; all other services have an
  empty bounding and ambient capability set. The passive unit permits only
  `AF_UNIX`, receive-only `AF_PACKET`, and the `AF_NETLINK` needed for bounded
  kernel-neighbour reads.
- Installed `systemd-analyze security` reports `1.3 OK` for passive observation,
  `1.2 OK` for the collector scheduler, and `2.7 OK` for backup, recovery, and
  nmap import. The passive seccomp policy permits `capset` solely so iproute2 can
  drop inherited `CAP_NET_RAW`; a matching installed-sandbox execution passed.
- Secret providers require exact regular non-symlink files with mode `0600`,
  or bounded helper stdin/stdout contracts. Artifact/report/backup publication
  uses owned directories, `0600` files, atomic replacement, and path/symlink
  confinement.
- Installation and removal were exercised under a clean `DESTDIR`. Removal
  targets exact CodexNet files and preserves the protected nmap script/results,
  cron state, Docker/Scanopy state, and retained customer data.

Production installation on the target ARM64 Pi now verifies `/opt/field-discovery`
as `root:root 0755`, configuration as `root:field-discovery 0640`, secrets as
`field-discovery:field-discovery 0600`, the state root as
`field-discovery:field-discovery 0750`, and SQLite/database sidecars as `0600`.
Recovery, passive observation, the idle deny-by-default collector scheduler,
and backup timer run under their dedicated unprivileged account. The external
nmap tree remains root-only pending an explicitly approved read-access design;
G6 records this as an open gate rather than weakening the protected boundary.

## Network, authentication, and target controls

- Active nmap remains an explicit operator action and is never scheduled by
  CodexNet. Approved-interface/range checks run before invocation.
- Collector orchestration rejects targets outside configured private ranges,
  bounds timeouts/retries/concurrency, and isolates individual failures.
- Production scheduled orchestration is installed as a hardened service. SNMP,
  AD, and SSH use concrete approved IPv4 targets; UniFi requires an HTTPS URL
  pinned to the same literal approved IPv4, preventing DNS target drift.
- Real disposable loopback SNMPv2c, authPriv SNMPv3, and strict-host-key SSH
  transports passed. SNMPv2c requires explicit opt-in; no community is guessed.
- SSH executes an exact read-only device-command and session-control allowlist.
  Configuration/write commands and hidden enable-mode transitions are refused.
- UniFi permits POST only to the exact modern or legacy login endpoint and GET
  without a body elsewhere. All near-miss/write methods fail before transport.
- TLS verification is the default. UniFi self-signed exceptions are endpoint
  specific; AD detection/collection uses certificate-required LDAPS or an
  existing Kerberos cache. Plain LDAP requires explicit configuration.

## Customer-data and deletion audit

The data inventory and lifecycle are defined in `docs/threat-model.md` and the
closeout procedure in `docs/operator-guide.md`. It accounts separately for:

- SQLite plus WAL/SHM files;
- sanitized raw artifacts and metadata;
- reports, diagrams, JSON, DOCX internals, and temporary conversions;
- database backups and restore candidates;
- journald output and diagnostic files;
- the externally owned nmap result tree; and
- credentials held only by configured providers.

Retention preview/apply parity, low-disk pause, exact-path pruning, backup and
restore verification, symlink refusal, failed-publication cleanup, and
install/remove preservation all pass. Closeout documentation explicitly forbids
claiming deletion while any database, backup, report, journal, or external nmap
copy remains.

Seeded secrets and prohibited AD content are absent from logs, process argument
models, SQLite exports, artifacts, backups, JSON, Mermaid/SVG, filenames, and
every DOCX ZIP member. DOCX validation rejects external relationships and
missing/broken content before a report can be declared upload-ready.
