[performance-soak.md](../performance-soak.md). The seven-day replay ran 168 virtual hourly cycles
in an auto-cleaned temporary directory. It used only sanitized fixtures and local application APIs;
no customer target, live collector, active scan, service install/restart, credential, or Scanopy
database was accessed.

Artifact count is exact and linear: 329 successful synthetic collector calls produced one bounded

on the next cycle. Daily pipeline recreation drained completely. Midpoint repository reopen proved
both transaction rollback and one-time interrupted-run recovery. Existing packaging tests verify
the persistent timers and boot ordering that run recovery before passive/import/backup writers.

## Protected coexistence comparison

The harness read only fixed Docker container state/health/restart metadata, the protected nmap
script checksum, crontab/timer state, and standard cron filenames before and after. Both snapshots
were identical:

- protected script SHA-256 remained
  `09bfdfd6d034c38882dfddf7cb648d64fc326fcf164f4c68ae49cb103eb2e526`;
- all three Scanopy containers remained running/healthy with the expected restart policy;
- user and root `crontab -l` each returned the exact no-crontab condition;
- the matching systemd-timer query and privileged filename-only standard-cron search found no
  nmap/network-discovery entry.

The generic diagnostic snapshot classified schedule inspection as a warning because `rg` is not on
the fixed system service PATH on this host. A supplemental read-only privileged search using the
installed absolute `rg` path returned no match. This does not treat unreadable state as proof of
absence: the user/root crontabs, timers, and standard cron directories were each checked directly.
No protected content, schedule, container, service, or database was changed.
